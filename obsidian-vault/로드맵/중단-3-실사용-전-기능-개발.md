# 중단 3 — 실사용 전 기능 개발

## 무엇을 멈추나

실제 수강생이 플랫폼을 써보기 전에 새 기능(대시보드, 통계, 추천, 알림 등)을 추가하는 것.

## 근거 (풀어서)

첫째, **지금의 기능 아이디어는 전부 가설이다.** 플랫폼은 아직 실제 수강생 0명이 썼다. "수강생이 이걸 원할 것"이라는 판단은 강사의 상상이고, 상상 기반 기능의 상당수는 빗나간다. 반면 첫 수업은 실사용 데이터 30건을 공짜로 주는 사용성 테스트다 — 어디서 막히고, 무엇을 다시 쓰고, 어떤 화면을 무시하는지가 전부 드러난다. 그 관찰 이후의 개발은 명중률이 완전히 다르다.

둘째, **기능은 추가하는 순간부터 유지 비용이 발생한다.** 코드가 늘면 수업 전 검증할 표면적이 늘고([[중단-2-수업-직전-시스템-변경]]), 버그 확률이 늘고, 초보 수강생이 헤맬 화면이 늘어난다. 이 플랫폼의 설계 철학이 "클릭을 줄이는 것"인데, 기능 추가는 대개 그 반대로 작용한다.

셋째, **대시보드가 특히 그렇다.** 아이디어 공유 대시보드는 데이터가 자산이 된 뒤에 가치가 생긴다. 데이터 3건으로 만든 대시보드는 초라해서 오히려 역효과다. 순서는 ① 동의 받고 데이터 축적([[무브-3-수업-데이터의-자산화]]) ② 3기수쯤 쌓이면 ③ 그 데이터가 실제로 보여주는 패턴에 맞춰 설계. 지금 DB에 원본이 온전히 보관되므로, 나중에 만들어도 잃는 것이 없다.

## 대신 할 것

- 기능 아이디어가 떠오르면 → 볼트에 노트 1장 (제목: 기능명, 본문: 어떤 관찰이 이걸 정당화할지)
- 첫 수업 후, 실제 관찰로 정당화된 상위 2개만 구현
- 예외는 수업 진행 자체를 막는 결함 수정뿐
