# 중단 2 — 수업 직전 시스템 변경

## 무엇을 멈추나

수업 24시간 안에 플랫폼 코드·모델·API 키·환경설정을 바꾸는 것. "간단한 수정이니까 괜찮다"는 예외 포함.

## 근거 (풀어서)

첫째, **모든 변경은 검증 시간을 요구하는데, 수업 직전에는 그 시간이 없다.** 이 프로젝트의 개발 기록만 봐도: 서버 재시작 후 구버전 프로세스가 포트를 점유한 채 살아남아 새 코드가 반영 안 된 일이 두 번, API 키가 한 글자(하이픈/언더스코어) 차이로 이틀간 동작하지 않은 일이 한 번, 데이터가 쌓인 뒤에야 드러난 직렬화 버그가 한 번 있었다. 전부 "바꾼 직후에는 멀쩡해 보였던" 변경이다. 개발 중에는 이런 발견이 성과지만, 수업 30분 전에는 재앙이다.

둘째, **실패의 비대칭성.** 직전 변경으로 얻는 것은 기껏해야 사소한 개선이고, 잃을 수 있는 것은 수업 전체와 기수의 신뢰다. 기대값 계산이 성립하지 않는 도박이다.

셋째, **강사는 수업 당일 인지 자원이 필요하다.** 직전에 시스템을 만진 날은 수업 중에도 "아까 그거 괜찮겠지"에 주의가 분산된다. 시스템을 동결하면 수강생에게 온전히 집중할 수 있다.

## 대신 할 것

- T-24시간 코드 동결 + [[무브-1-수업-리허설과-운영-런북]]의 리허설 체크리스트 수행
- 수업 중 발견한 개선 아이디어는 그 자리에서 고치지 말고 볼트에 노트로 적기
- 변경은 수업 다음 날, 여유 있을 때 몰아서
