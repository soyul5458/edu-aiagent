# 계정 삭제와 기록 보존은 "스냅샷 + SET NULL"로 분리한다

## 문제

관리자가 수강생 계정을 삭제하면 그 수강생의 아이디어까지 함께 사라졌다(FK `ON DELETE CASCADE`). 아이디어 데이터는 미래 대시보드의 자산이라 계정과 무관하게 보존돼야 한다.

## 헛짚은 것

처음 설계 때 "계정 삭제 = 그 사람 데이터 정리"로 묶어서 생각했다. 실제로는 **계정(접근 권한)과 기록(자산)은 수명이 다르다** — 수강생은 기수가 끝나면 떠나지만 기록은 남아야 한다.

## 해결

- FK를 `ON DELETE SET NULL`로 바꾸고, 작성 시점에 `author_name`을 스냅샷으로 저장 → 계정이 사라져도 갤러리에 이름과 기록이 그대로 남음
- SQLite는 제약조건 변경(ALTER)이 안 되므로 **테이블 재구축 마이그레이션**(새 테이블 생성 → 데이터 복사 → 교체)을 getDb()에 넣어 기존 DB도 자동 전환

## 다음에 쓸 규칙

- FK를 설계하는 순간 "부모가 삭제되면 무엇이 남아야 하는가?"를 먼저 답하라. CASCADE는 기본값이 아니라 의도적 선택이어야 한다
- 표시용 정보(이름 등)는 조인에만 의존하지 말고 기록 시점 스냅샷을 남겨라 — 원본이 사라져도 표시가 깨지지 않는다
- SQLite 제약 변경은 항상 재구축 패턴: 컬럼 존재 여부(PRAGMA table_info)로 구버전을 감지해 1회성 마이그레이션
