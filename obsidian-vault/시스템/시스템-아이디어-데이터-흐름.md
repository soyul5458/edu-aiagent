# 수강생 아이디어는 어디에 저장되는가

## 결론 한 줄

**원본은 DB(`data/platform.db`), 옵시디언은 파생 지식 레이어.** 대시보드는 DB에서 만들고, 통찰은 볼트에서 캔다.

## 데이터 흐름

```
수강생이 기획서 붙여넣기
  → AI 정제 (제목·카테고리·태그·요약 자동 생성)
  → data/platform.db 에 저장  ←←← 유일한 원본 (원본 기획서 + MVP 프롬프트 전문 포함)
      ├→ 갤러리 화면 (즉시 공유)
      ├→ [미래] 아이디어 공유 대시보드 ← 반드시 여기서 조회
      └→ 관리자 [옵시디언 내보내기] 버튼 → obsidian-vault/아이디어/*.md (파생물)
```

## 왜 DB가 원본이어야 하나

- 대시보드가 필요로 하는 것은 **집계와 질의**다: 카테고리별 수, 기간별 추이, 작성자별 목록, 검색. SQL 한 줄이면 되는 일을 마크다운 파일 뭉치로 하려면 파서를 새로 짜야 한다.
- 여러 수강생이 동시에 쓰는 웹 서비스의 저장소는 동시성·무결성을 보장해야 한다. SQLite는 그걸 하고, 파일 폴더는 못 한다.
- 옵시디언은 데이터베이스가 아니라 **한 사람(강사)의 사고 도구**다. 역할이 다르다.

## 옵시디언은 그럼 왜 연결하나

- 아이디어 300개를 SQL로 집계하면 "숫자"가 나오고, 볼트에서 링크 타고 읽으면 "다음 강의 아이템"이 나온다. 통찰 발굴은 볼트가 강하다.
- 미래의 AI 세션이 볼트를 맥락으로 읽는다 — 수강생들이 실제로 원한 것이 맥락에 들어 있으면 강의 개선·상품 기획의 질이 달라진다.
- 그래서 다리를 놓았다: 관리자 페이지 → [옵시디언 내보내기] → 아이디어 1건 = 노트 1장(카테고리 링크 포함).

## 보관 규칙

- 백업 = `data/` 폴더 복사 (수업 후 5분 루틴, [[무브-3-수업-데이터의-자산화]])
- 볼트의 아이디어 노트는 언제든 버튼으로 재생성 가능 — 볼트에서 지워져도 원본은 무사
- 반대로 DB를 지우면 복구 불가 — `data/`는 절대 손대지 않는다
